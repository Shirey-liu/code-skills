# MR / 任务经验精炼总结

## 角色
在 MR 合入后或任务做完后，把本次经验压成极短复盘，写入仓库 `docs/retros/`，避免堆砌长文档。

## 触发
用户说「总结这次」、MR 已合入/任务已完成并要求复盘时启动。

## 硬性约束
- **精炼**：每项不超过模板上限；禁止长篇背景、changelog、代码大段粘贴
- **只写事实**：只写本次真实遇到的点；没有冲突就写「无」
- **冲突/坑重点**：写清「解 A 时踩了什么、如何避免又影响 B」；强调不要改完一处又破坏别处

## 工作流

1. **收集上下文**（能读则读，不够再问一句）：
   - 分支名 / MR 标题 / TAPD 任务名
   - `git log`、关键 diff（只看本次范围）
   - 对话里已确认的坑与注意点
2. **起草复盘**（严格套模板，先给用户看）
3. **用户确认措辞后**再写入文件
4. **落盘路径**：`docs/retros/YYYY-MM-DD-<任务短名>.md`
   - 目录不存在则创建
   - `<任务短名>`：英文/拼音短横线，尽量短（如 `region-rights-pack`）
5. **不自动 commit**，除非用户明确要求提交

## 输出模板（必须 verbatim）

文件内容只保留下面结构，不要额外大标题或附录：

```text
任务名: [一句话]
主要做什么: [1-2 句]
解决了什么: [1-2 句]
注意点: [最多 3 条]
冲突/坑: [解决某个问题时遇到了什么；如何避免改完此处又影响其他内容。无则写「无」]
```

## 写法要求

| 字段 | 要求 |
|------|------|
| 任务名 | 一句话，可读作任务标题 |
| 主要做什么 | 1–2 句，说清范围，不写实现细节 |
| 解决了什么 | 1–2 句，说清结果/用户侧收益 |
| 注意点 | 最多 3 条短句；可编号或 `- ` |
| 冲突/坑 | 写「问题 → 影响面 → 以后怎么防」；无则「无」 |

## 交互示例

用户: 总结这次  
AI: 根据分支/MR 起草模板内容 → 确认后写入 `docs/retros/2026-07-16-xxx.md` → 告知路径，询问是否要 git 提交

## 反例（禁止）

- 贴完整 commit list、大段 diff、OpenSpec 全文
- 「注意点」超过 3 条或写成教程
- 没有冲突也硬编故事
